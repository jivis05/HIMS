# WeaveHealth HIMS - Backend Architecture & Structure

This document outlines the architecture, directory structure, and technology stack of the Backend portion of the WeaveHealth Hospital Information Management System (HIMS).

## 1. Technology Stack

*   **Core Environment:** Node.js
*   **Web Framework:** Express.js
*   **Database:** MongoDB
*   **ODM (Object Data Modeling):** Mongoose
*   **Authentication & Security:** JWT (JSON Web Tokens), bcryptjs (hashing), speakeasy (2FA/TOTP)
*   **Validation:** express-validator
*   **Logging:** Winston & Morgan
*   **Documentation:** Swagger (swagger-jsdoc, swagger-ui-express)
*   **Testing:** Jest & Supertest

## 2. Top-Level Directory Structure

The backend application is contained within the `backend/` directory and follows an MVC (Model-View-Controller) inspired architectural pattern for APIs.

```text
backend/
├── config/                # Database connections and third-party service configurations
├── controllers/           # Business logic and request handling
├── logs/                  # Application runtime log files (generated by Winston)
├── middleware/            # Custom Express middleware (Auth, Error handling, RBAC)
├── models/                # Mongoose Database Schemas
├── routes/                # API route definitions and endpoint mapping
├── scripts/               # Utility scripts (e.g., database seeding)
├── testcases/             # Automated test suites (Jest/Supertest)
├── utils/                 # Helper functions and shared utilities
├── .env                   # Environment variables
├── package.json           # Dependencies and NPM scripts
└── server.js              # Application entry point
```

## 3. Deep Dive into the Architecture

The backend is modularized based on healthcare domains (e.g., Lab, Appointments, Pharmacy, Blood Bank). Each domain generally has a Model, a Route file, and a Controller.

### 3.1. Entry Point (`server.js`)
*   Initializes the Express application.
*   Connects to MongoDB using configurations from the `config/` directory.
*   Applies global middleware (CORS, Helmet, Body Parser, Morgan logging).
*   Mounts the main route handlers.
*   Starts the HTTP server.

### 3.2. Models (`models/`)
Defines the structure, validation, and relationships of data in MongoDB using Mongoose schemas. Key models include:
*   `User.model.js`: Stores all user accounts (Patients, Doctors, Admins) and handles RBAC roles.
*   `Organization.model.js`: Supports multi-tenancy by representing different hospital branches or facilities.
*   `Appointment.model.js` & `LabAppointment.model.js`: Manages scheduling.
*   `Prescription.model.js` & `LabReport.model.js`: Clinical records.
*   `Inventory.js`, `Bed.js`, `BloodStock.js`, `Invoice.js`: Operational and financial records.
*   `Consent.model.js`: Manages patient data privacy and consent forms.

### 3.3. Controllers (`controllers/`)
Contains the core business logic. They receive the request from the router, interact with the database (via Models), process the data, and send back the HTTP response.
*   `auth.controller.js`: Handles login, registration, and 2FA setup.
*   `appointment.controller.js`: Booking, canceling, and retrieving appointments.
*   `lab.controller.js` & `labReport.controller.js`: Lab workflows.
*   `org.controller.js`: Organization management (multi-tenancy).

### 3.4. Routes (`routes/`)
Maps HTTP methods (GET, POST, PUT, DELETE) and endpoints to their respective Controller functions. Also applies specific middleware (like authentication guards or validation chains) to individual routes.
*   `auth.routes.js`
*   `user.routes.js`
*   `appointment.routes.js`
*   *(and many others corresponding to the controllers)*

### 3.5. Middleware (`middleware/`)
Functions that run between receiving a request and executing the controller logic.
*   **Authentication (`auth.js` or similar):** Verifies JWT tokens present in the request headers to ensure the user is logged in.
*   **Role-Based Access Control (RBAC):** Checks if the authenticated user has the required role (e.g., only 'SuperAdmin' can access certain routes).
*   **Data Scoping (Organization Isolation):** Ensures that queries are restricted to the user's specific organization/hospital facility to maintain data privacy in a multi-tenant environment.
*   **Validation:** Uses `express-validator` to ensure incoming request bodies match expected formats before hitting controllers.

### 3.6. Utils (`utils/`)
Shared helper functions to keep controllers clean.
*   Error formatting helpers.
*   Pagination standardizers.
*   Data transformers.

## 4. Security Flow
1.  **Login:** User provides credentials -> `auth.controller` verifies password via `bcrypt` -> Checks 2FA (if enabled) via `speakeasy` -> Generates and returns a JWT.
2.  **Subsequent Requests:** Frontend sends JWT in `Authorization: Bearer <token>` header -> `middleware` intercepts, decodes token, and attaches the `user` object to the Express `req` object.
3.  **Authorization:** Route-specific middleware checks if `req.user.role` is allowed to access the endpoint.

## 5. Available Scripts
Defined in `package.json`:
*   `npm start`: Runs the server normally.
*   `npm run dev`: Runs the server with `nodemon` for auto-reloading during development.
*   `npm test`: Executes the Jest test suites to validate business logic and API endpoints.
